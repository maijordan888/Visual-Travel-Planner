# UI 建立備選方案 Workflow

本 workflow 用於使用者提供多個未定選項，例如晚餐二選一、下午行程二選一。若使用者要求 UI-only，禁止直接建立 `pending_options` store 資料。

## 重要限制

- 只能使用畫面上的 `新增其他備選方案`、`切換採用`、地圖搜尋與確認按鈕。
- 不得直接寫入 `nodesByDay`、`options`、localStorage 或 API。
- 如果 UI 目前無法建立「沒有主選項的 pending_options 節點」，需誠實回報，並用 UI 可支援的備選節點行為完成最接近結果。

## 從已採用節點新增備選

1. 先依 `.agent/workflows/add-place-with-google-placepicker.md` 加入目前最可能採用的主選項。
2. 在該節點上點擊 `新增其他備選方案`。
3. 點擊 `開啟地圖/交由 AI 推薦`。
4. 搜尋備選地點並點選 Google 建議結果。
5. 點擊 `確認加入`。
6. 回到節點後展開備選方案列表。
7. 檢查新加入的地點是否成為備選，或是否被 App 自動切成目前採用。
8. 如果 App 自動採用新地點，使用 `切換採用` 把原本主線切回來。
9. 重複步驟 2-8 加入其他備選。

## 未定選項的處理規則

- 如果使用者明確指定「不要 confirmed」，但 UI 只能在有主選項時掛備選，先選一個最合理的暫定主選項，並在節點備註或最終回覆中標明仍待決定。
- 不要把同一時段的多個晚餐、景點直接新增成連續 confirmed 節點，除非使用者同意這樣呈現。
- 若 UI 操作會改變採用狀態，完成後必須再次確認主選項與備選列表。

## 驗證

- 同一個未定時段的備選出現在同一節點的 options / 備選列表中。
- confirmed 主選項符合使用者目前最可能採用的安排。
- 備選地點同樣透過 Google PlacePicker 加入，保留圖片與地點資料。
- 沒有把互斥選項散成多個連續 confirmed 景點。
- 最終回覆需說明哪些項目仍是備選，哪些是目前採用。
